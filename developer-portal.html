Re-imagining

Developer Portal — API Documentation

Removed the authentication barrier to API documentation so developers could evaluate shipping integration capabilities — and decide if Dispatch was the right fit — before creating an account.

Problem

▼

Developers evaluating the carrier's shipping APIs faced a two-gate authentication process: they had to create and verify an account just to access the API documentation, and then authenticate a second time to activate production API keys. The first gate was the friction point — developers wanted to understand available capabilities, review API specifications, assess integration complexity, and determine whether our APIs fit their use case before committing to account creation.

This is the standard developer expectation: documentation should be publicly discoverable. Requiring sign-up before a developer could read a single endpoint description meant many qualified integration partners were lost before they ever reached the evaluation stage.

Constraints

▼
  • Authentication on the documentation portal was an existing security decision — removing it entirely required stakeholder alignment across product, legal, and security
  • Abuse protection still had to be maintained: the concern was that open documentation could be scraped or used by bad actors
  • The second authentication gate (for production API key activation) was intentional and non-negotiable — only the first gate was in scope
  • The "Try it" / sandbox experience had to remain behind authentication to prevent unauthenticated API calls being made against real systems
  • The redesign had to preserve the existing information architecture — this was a friction-reduction project, not a full rebuild

My Contribution

▼

Stakeholder Alignment

Led discussions with product, legal, and security to challenge the assumption that documentation access required authentication. The key insight was separating "access to docs" from "access to APIs" — two distinct privileges that had been conflated.

Abuse Protection Design

Partnered with engineering to implement reCAPTCHA on the documentation portal as a lightweight, frictionless abuse deterrent. This satisfied the security requirement without requiring sign-in — humans see documentation, bots do not.

Access Tiering

Defined a two-tier access model: documentation publicly available with reCAPTCHA protection; sandbox and API key activation behind authentication. The "Try it" button was preserved but surfaced a sign-in prompt only when clicked — not before a developer could read anything.

Developer Journey Mapping

Mapped the developer evaluation journey — discovery, assessment, prototype, activate — and identified the first gate as occurring before step 1 was even complete. Removing it aligned the experience with developer expectations and industry norms.

How It Worked

▼

After the redesign, developers landing on the portal could immediately browse all API documentation — endpoints, parameters, response schemas, error codes, and integration guides — without signing in. reCAPTCHA ran invisibly in the background, providing bot protection without visible challenge in most cases.

The "Try it" sandbox feature, and the API key management section, remained behind authentication. When a developer clicked "Try it" on any endpoint, a modal prompted sign-in — at which point they had already assessed fit and had a reason to create an account. The authentication barrier was moved from the discovery stage to the intent stage, where it belongs.

Outcome

▼

The new experience aligned documentation access with developer expectations and improved self-service discovery of shipping integration capabilities. Removing the first authentication gate reduced drop-off at the top of the developer funnel. Account activations from organic documentation discovery increased, and the integration team reported fewer "we had no idea what your API looked like" conversations in sales-led deals — a signal that developers were arriving at conversations already informed.
