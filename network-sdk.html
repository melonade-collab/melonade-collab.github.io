Re-imagining

Network Status SDK

A consumer-facing SDK and mobile experience for checking network signal quality, reporting coverage issues, and surfacing crowdsourced outage data — turning every device into a passive coverage sensor.

Problem

▼

Customers experiencing network issues had no self-serve way to know whether the problem was with their device, their local environment, or an actual network outage. The only options were calling support — which was expensive and slow — or simply waiting and hoping. Meanwhile, the carrier's own network monitoring tools detected outages reactively, after enough calls came in to hit a threshold.

The problem had two sides: customers lacked visibility into what was happening, and the carrier was collecting coverage intelligence the hard way — through support interactions rather than from the devices actually on the network.

Constraints

▼
  • SDK had to be embedded in the carrier's existing consumer app — it wasn't a standalone app
  • Network diagnostics had to use only on-device APIs (signal strength, network type, latency) — no deep carrier-side data pipe for individual customers
  • Reports had to be anonymous enough to protect individual location privacy, while still being geographically precise enough to be actionable
  • The crowdsourcing threshold needed careful tuning: too low and noise becomes outages; too high and real outages don't surface quickly enough
  • Known outage data came from the carrier's NOC — the SDK had to consume that via API without exposing internal infrastructure
  • Speed test traffic had to be metered carefully to avoid consuming customer data allowances unintentionally

My Contribution

▼

Crowdsource Model

Designed the threshold logic: user-submitted reports within a defined geographic cell above a volume + time-window threshold automatically elevated to "known outage" status and triggered NOC review. Gave customers a visible counter so they could see whether their issue was isolated or shared.

SDK Feature Set

Scoped the SDK around the four most-searched support topics: signal quality, outage status, issue reporting, and speed testing. Kept the API surface small so embedding teams could integrate quickly without becoming experts in telecom diagnostics.

Outage UX

Designed a map + list view for known outages, surfacing estimated resolution times from the NOC and allowing customers to subscribe to resolution notifications — converting a frustrating wait into a managed expectation.

Report-to-Outage Feedback Loop

Built a status lifecycle into the reporting flow: submitted → under investigation → added to known outage → resolved. Customers who submitted reports received push notifications at each state change, turning a one-way report into a two-way conversation with the network.

How It Worked

▼

On the Status tab, the SDK read signal strength and network type from the device's telephony API and displayed them alongside a latency measurement from a lightweight ping to the carrier's nearest edge node. A rolling sparkline showed signal quality over the current session.

When a customer submitted a report, the SDK attached anonymized GPS coordinates rounded to a ~500m grid cell and the current network diagnostic snapshot. If that grid cell accumulated reports above the threshold within a rolling time window, the system automatically created a candidate outage event and surfaced it in the known outages list — usually before a single customer had called support.

The Speed Test ran a series of short bursts to minimize data usage while still producing a meaningful download and upload estimate. Results were stored locally for trend comparison across sessions. Customers could also see whether their measured speed matched what their plan promised — a gentle but clear signal when throttling was happening.

Outcome

▼

Reduced support call volume for coverage complaints by giving customers a self-serve answer before they reached for the phone. The crowdsourced outage detection consistently surfaced issues 8–12 minutes faster than traditional NOC monitoring, improving both mean time to detect and the customer experience of being heard. The report-to-notification loop became one of the highest-rated features in post-interaction surveys — customers didn't mind a problem nearly as much when they knew the carrier already knew about it.
